## 1. Концепция

**Design Lint** — плагин для аудита и массового исправления дизайн-файлов Figma. Три основных режима работы: **Типографика**, **Цвета**, **Эффекты**. Каждый режим сканирует файл, выявляет элементы без привязки к стилям/переменным (Variables), суммаризирует найденные значения и позволяет массово переназначить их.

Дополнительно плагин обнаруживает стилевые оверрайды в инстансах компонентов для их сброса к значениям мастер-компонентов, при этом не трогая контентные изменения (текст, иконки) и показывая изменения изображений отдельной группой с подтверждением.

---

## 2. Проблема

Дизайн-файлы «загрязняются» со временем:

- Шрифтовые гарнитуры (font families) меняются по ходу проекта, но старые остаются на десятках экранов
- Текстовые слои создаются без привязки к TextStyle — числовые параметры (кегль, интерлиньяж) «гуляют»
- Цвета хардкодятся вместо привязки к Variables или Color Styles
- Эффекты (тени, блюры) копируются руками и расползаются
- Инстансы компонентов содержат стилевые оверрайды, нарушающие единообразие дизайн-системы
- Ручная проверка невозможна в файлах с сотнями экранов

| Преимущество | Описание |
|-------------|----------|

| **Скрытые элементы** | Корректная обработка hidden layers (`visible: false`) — включаются в сканирование, но помечаются; пользователь решает |
| **Изображения в инстансах** | Отдельная группа с подтверждением, не сбрасываются автоматически |
| **Три режима** | Типографика, Цвета, Эффекты — вместо одного универсального экрана |

## 4. Функциональные требования

### FR-01: Режим «Типографика» (Typography Mode)

#### FR-01.1: Замена шрифтовой гарнитуры (Font Replacement)

Аналог встроенной функции Figma «Replace missing font», но работающий с любыми шрифтами, не только с отсутствующими.

**Функционал:**
- Сканирование всех текстовых слоёв в выбранной области (scope)
- Суммаризация найденных шрифтовых гарнитур (font families) с количеством использований
- Для каждой гарнитуры — список используемых начертаний (font styles): Regular, Bold, Semi Bold и т.д.
- Возможность выбрать целевую гарнитуру из списка доступных шрифтов (с поиском)
- Маппинг начертаний: при замене `Inter Bold` → `Roboto Bold`, если `Roboto Bold` не существует — предложить ближайшее (`Roboto 700`)
- Массовая замена одним кликом
- Preview: показать количество затронутых слоёв перед применением

**Обработка mixed fonts:**
- TextNode может содержать несколько шрифтов внутри одного слоя (`fontName === figma.mixed`)
- Использовать `getStyledTextSegments(['fontName'])` для получения всех сегментов
- Замена только сегментов с целевой гарнитурой, остальные не трогать
- Обязательная загрузка всех шрифтов через `loadFontAsync()` перед модификацией

**Техническая реализация:**
```typescript
// Получение всех уникальных font families
interface FontUsage {
  family: string;
  styles: Map<string, number>; // style → count of text segments
  nodeIds: string[];           // ID нод, использующих этот шрифт
  isInInstance: boolean;       // хотя бы одна нода внутри Instance
}

// Маппинг начертаний при замене гарнитуры
interface StyleMapping {
  sourceStyle: string;   // "Bold"
  targetStyle: string;   // "700" или "Bold"
  autoMatched: boolean;  // совпало автоматически или подобрано ближайшее
}
```

#### FR-01.2: Аудит текстовых стилей (Text Style Audit)

**Сканирование:**
- Поиск текстовых слоёв без привязки к TextStyle (`textStyleId === ''`)
- Для каждого слоя — сбор числовых параметров:
  - `fontSize` — кегль
  - `lineHeight` — межстрочный интервал (line height)
  - `letterSpacing` — трекинг (letter spacing)
  - `paragraphSpacing` — межабзацный интервал
  - `fontName` — гарнитура + начертание
  - `fontWeight` — вес шрифта
- Обработка `figma.mixed` для каждого параметра — если значение mixed, разбить по сегментам

**Суммаризация:**
- Группировка текстовых слоёв по комбинации параметров: `{family}:{style}:{size}:{lineHeight}`
- Для каждой группы: количество слоёв, список ID
- Поиск ближайшего существующего TextStyle по параметрам (fuzzy matching):
  - Совпадение по fontSize с допуском ±1px
  - Совпадение по lineHeight с допуском ±2px
  - Совпадение по fontFamily — точное
- Подсказка: «Похож на стиль `Body/Regular` (отличие: fontSize 15 вместо 16)»

**Массовая операция:**
- Пользователь выбирает группу несвязанных текстовых слоёв
- Выбирает целевой TextStyle или создаёт новый из текущих параметров
- «Применить» → привязка `textStyleId` ко всем слоям группы
- Альтернатива: массовое изменение числовых параметров без привязки к стилю (например, все `fontSize: 15` → `fontSize: 16`)

**Список параметров для массовой замены:**

| Параметр | Тип | Примечание |
|----------|-----|------------|
| `fontSize` | number | Кегль |
| `lineHeight` | `{ value: number, unit: 'PIXELS' \| 'PERCENT' \| 'AUTO' }` | Интерлиньяж |
| `letterSpacing` | `{ value: number, unit: 'PIXELS' \| 'PERCENT' }` | Трекинг |
| `paragraphSpacing` | number | Межабзацный |
| `fontName` | `{ family: string, style: string }` | Гарнитура + начертание |
| `textStyleId` | string | Привязка к стилю |

### FR-02: Режим «Цвета» (Color Mode)

#### FR-02.1: Аудит привязки цветов

**Сканирование:**
- Проверка всех нод в scope на наличие привязки fills/strokes к:
  - Color Variables (через `boundVariables.fills`)
  - Fill Styles (через `fillStyleId`)
  - Stroke Styles (через `strokeStyleId`)
- Для непривязанных — извлечение значения цвета
- Поддержка типов fill:
  - `SOLID` — основной, полная поддержка
  - `GRADIENT_LINEAR`, `GRADIENT_RADIAL`, `GRADIENT_ANGULAR`, `GRADIENT_DIAMOND` — показывать, но не автоматически матчить
  - `IMAGE` — пропускать в цветовом анализе
  - `VIDEO` — пропускать

**Суммаризация:**
- Группировка по значению цвета (HEX + opacity)
- Для каждой группы: количество использований, ID нод
- Разделение на fills и strokes
- Поиск ближайшей Variable или Style:
  - Color distance (Delta E в OKLCH или простой Euclidean в RGB)
  - Порог совпадения настраиваемый (по умолчанию: ΔE ≤ 3)
  - Подсказка: «#3366CB → Похож на `color/primary/500` (#3366CC), расстояние: 0.4%»

#### FR-02.2: Массовая замена цветов

**Операции:**
- Выбрать группу одинаковых непривязанных цветов → привязать к Variable
- Выбрать группу → привязать к Fill/Stroke Style
- Выбрать группу → заменить значение цвета на другое (без привязки)
- Заменить одну Variable/Style на другую (свап)

**Техническая реализация привязки fill к Variable:**
```typescript
// Паттерн: копия → модификация → присвоение
const fillsCopy = [...node.fills] as Paint[];
fillsCopy[0] = figma.variables.setBoundVariableForPaint(
  fillsCopy[0],
  'color',
  targetVariable
);
node.fills = fillsCopy;
```

**Техническая реализация привязки к Style:**
```typescript
// Для dynamic-page
await node.setFillStyleIdAsync(targetStyle.id);
```

### FR-03: Режим «Эффекты» (Effects Mode)

#### FR-03.1: Аудит эффектов

**Сканирование:**
- Проверка всех нод на наличие effects (`node.effects`)
- Проверка привязки к Effect Styles (`effectStyleId`)
- Типы эффектов:
  - `DROP_SHADOW` — внешняя тень
  - `INNER_SHADOW` — внутренняя тень
  - `LAYER_BLUR` — размытие слоя
  - `BACKGROUND_BLUR` — размытие фона

**Суммаризация:**
- Группировка по сигнатуре эффекта: `{type}:{color}:{offset}:{blur}:{spread}`
- Для каждой группы: количество, ID нод
- Поиск ближайшего Effect Style

#### FR-03.2: Массовая замена эффектов

- Привязка к существующему Effect Style
- Замена параметров эффекта (blur radius, offset, color, spread)
- Привязка цвета эффекта к Variable:

```typescript
const effectsCopy = [...node.effects] as Effect[];
effectsCopy[0] = figma.variables.setBoundVariableForEffect(
  effectsCopy[0],
  'color',
  shadowColorVar
);
node.effects = effectsCopy;
```

### FR-04: Обработка инстансов (Instance Handling)

#### FR-04.1: Пропуск инстансов при массовой замене

- По умолчанию все ноды типа `INSTANCE` и их дочерние элементы **пропускаются** при массовой замене
- Переключатель «Включить инстансы» (`Include instances`) — выключен по умолчанию
- При включении — предупреждение: «Изменение стилей в инстансах создаёт оверрайды, отличающиеся от мастер-компонента»

**Определение принадлежности к инстансу:**
```typescript
// Рекурсивная проверка: нода или любой из её предков — INSTANCE
function isInsideInstance(node: SceneNode): boolean {
  let current: BaseNode | null = node;
  while (current) {
    if (current.type === 'INSTANCE') return true;
    current = current.parent;
  }
  return false;
}
```

#### FR-04.2: Аудит оверрайдов в инстансах (Instance Overrides Audit)

Отдельная подвкладка в каждом режиме — «Оверрайды в инстансах» (Instance Overrides).

**Что проверяется:**

| Категория | Стилевой оверрайд (показываем) | Контентный оверрайд (пропускаем) |
|-----------|-------------------------------|----------------------------------|
| Текст | fontFamily, fontSize, lineHeight, letterSpacing, fontWeight, textStyleId | `characters` (текстовое содержимое) |
| Цвет | fills (color), strokes (color), fillStyleId, strokeStyleId | — |
| Эффекты | effects, effectStyleId | — |
| Изображения | — | fills с `type: IMAGE` → **отдельная группа** |

**Определение оверрайда:**
- Сравнение свойства ноды в инстансе с таким же свойством в соответствующей ноде мастер-компонента
- Используем `instance.getMainComponentAsync()` для получения мастера
- Рекурсивное сопоставление по `node.name` или позиции в дереве

**Действия:**
- «Сбросить выбранные» — удаление стилевых оверрайдов для выбранных нод
- «Сбросить все» — удаление всех стилевых оверрайдов (кроме контентных и изображений)
- `instance.removeOverrides()` — нативный метод Figma, но он сбрасывает ВСЕ оверрайды; для селективного сброса — ручное присвоение значений из мастера

#### FR-04.3: Отдельная группа — Изображения в инстансах

- Fills с `type: IMAGE` внутри инстансов, отличающиеся от мастер-компонента
- Показываются в отдельной группе «Изменённые изображения (требует подтверждения)»
- Каждый элемент — с чекбоксом, ни один не выбран по умолчанию
- Только при явном выборе пользователем — сброс к изображению из мастера

### FR-05: Область сканирования (Scan Scope)

#### FR-05.1: Три уровня scope

| Scope | Описание | Поведение |
|-------|----------|-----------|
| **Selection** | Только выделенные ноды и их дочерние | По умолчанию |
| **Page** | Все ноды текущей страницы | Предупреждение о возможной длительности |
| **Document** | Все страницы документа | Предупреждение + требуется `loadAsync()` для каждой страницы |

#### FR-05.2: Сканирование только по кнопке

- Плагин **никогда** не запускает сканирование автоматически
- Кнопка «Сканировать» (Scan) — единственный триггер
- При смене scope / режима / переключении вкладок — предыдущие результаты сохраняются до нового сканирования
- Нет слежения за `selectionchange` для автоматического ресканирования

#### FR-05.3: Исключения

- Ноды с pluginData `lint-ignore: true` — пропускаются при сканировании
- Контекстное меню (правый клик по ноде в результатах) → «Игнорировать этот элемент»

### FR-06: Производительность (Performance)

> **Ключевое конкурентное преимущество.** Конкуренты виснут — мы нет.

#### FR-06.1: Чанковая обработка (Chunked Processing)

Figma plugin main thread — **однопоточный** и разделяет thread с UI Figma. Любая тяжёлая операция замораживает весь интерфейс. Официальная документация Figma рекомендует разбивать работу на чанки с yield через `setTimeout`.

**Реализация:**
```typescript
// Утилита для yield в main thread
function yieldToMain(): Promise<void> {
  return new Promise(resolve => setTimeout(resolve, 0));
}

// Константы чанкования
const CHUNK_SIZE = 50;        // Нод за один чанк
const YIELD_INTERVAL_MS = 0;  // setTimeout(0) — минимальный yield

// Обход дерева с чанкованием
async function traverseWithChunks(
  roots: SceneNode[],
  processor: (node: SceneNode) => void,
  onProgress: (processed: number, total: number) => void
): Promise<void> {
  // Шаг 1: Собрать все ноды в плоский список (быстрая операция)
  const allNodes: SceneNode[] = [];
  function collect(node: SceneNode) {
    allNodes.push(node);
    if ('children' in node) {
      for (const child of node.children) {
        collect(child);
      }
    }
  }
  for (const root of roots) collect(root);

  // Шаг 2: Обработка чанками
  for (let i = 0; i < allNodes.length; i++) {
    processor(allNodes[i]);

    // Yield каждые CHUNK_SIZE нод
    if (i % CHUNK_SIZE === 0) {
      onProgress(i, allNodes.length);
      await yieldToMain();
    }
  }

  onProgress(allNodes.length, allNodes.length);
}
```

#### FR-06.2: Прогрессивная отдача результатов

- Каждый yield — отправка промежуточных результатов в UI через `postMessage`
- UI обновляется по мере поступления данных
- Пользователь видит:
  - Прогресс-бар: `Сканирование: 1 247 / 3 891 нод (32%)`
  - Уже найденные группы значений появляются в списке
  - Можно начать анализировать результаты, не дожидаясь завершения

```typescript
// Отправка промежуточных результатов
if (i % CHUNK_SIZE === 0) {
  figma.ui.postMessage({
    type: 'scan-progress',
    processed: i,
    total: allNodes.length,
    partialResults: currentResults, // уже собранные данные
  });
  await yieldToMain();
}
```

#### FR-06.3: Кнопка «Стоп» (Cancel Scan)

- Во время сканирования кнопка «Scan» меняется на «Stop»
- При нажатии — прерывание цикла обработки
- Уже собранные результаты сохраняются и доступны для работы

```typescript
let scanCancelled = false;

figma.ui.onmessage = (msg) => {
  if (msg.type === 'cancel-scan') {
    scanCancelled = true;
  }
};

// В цикле обработки
for (let i = 0; i < allNodes.length; i++) {
  if (scanCancelled) {
    scanCancelled = false;
    break; // Прерывание, partialResults уже в UI
  }
  // ... обработка
}
```

#### FR-06.4: Оптимизация — пропуск неактуальных нод

В зависимости от режима, пропускаем ноды, которые точно не содержат релевантных данных:

| Режим | Пропускаем |
|-------|-----------|
| Typography | Ноды без `type === 'TEXT'` (но собираем для вложенности) |
| Color | Ноды без `fills` и `strokes` (например, `GROUP`) |
| Effects | Ноды без `effects` |

### FR-07: Обработка скрытых элементов (Hidden Layers)

#### FR-07.1: Проблема

В Figma ноды с `visible: false` (скрытые слои) полностью доступны через API — их свойства можно читать и модифицировать. Однако при массовой замене конкуренты не учитывают скрытые слои, что приводит к неожиданным результатам:

- Скрытые слои — часто это альтернативные состояния компонентов (hidden toggle)
- Изменение стиля скрытого слоя может сломать компонент при переключении состояния
- Пользователь не видит скрытый слой и не ожидает, что он будет изменён

#### FR-07.2: Решение

1. **При сканировании:** скрытые ноды включаются в результаты, но помечаются тегом `👁 Скрытый` (Hidden)
2. **В UI:** отдельный фильтр «Показать скрытые элементы» — включён по умолчанию при просмотре, но при массовом действии скрытые по умолчанию **не выбраны**
3. **При массовой замене:** скрытые ноды включаются только если пользователь явно поставил чекбокс «Включить скрытые слои» или выбрал их вручную
4. **Навигация:** при клике на скрытую ноду в списке — `scrollAndZoomIntoView` + подсветка в Layers panel (нода будет видна как серая)

```typescript

  nodeId: string;
  nodeName: string;
  nodePath: string;          // "Page / Frame / Group / Node"
  isHidden: boolean;         // visible === false
  isInsideInstance: boolean;  // внутри Instance
  // ... специфичные для режима поля
}
```

### FR-08: Навигация и взаимодействие

#### FR-08.1: Навигация к ноде

- Клик по элементу в списке результатов → `figma.viewport.scrollAndZoomIntoView([node])`
- Выделение ноды → `figma.currentPage.selection = [node]`
- Для нод на другой странице (при scope Document) → `setCurrentPageAsync` + `loadAsync` + scroll

#### FR-08.2: Группировка результатов

В каждом режиме результаты группируются по значению:

**Typography:**
```
▼ Inter / Regular / 16px / LH: 24px — 47 слоёв
  ├── 42 слоя — не привязаны к TextStyle
  │   → Похож на стиль "Body/Regular" [Привязать]
  └── 5 слоёв — привязаны к "Body/Regular" ✓
▼ Inter / Regular / 15px / LH: 22px — 12 слоёв
  └── 12 слоёв — не привязаны
      → Ближайший: "Body/Regular" (16/24) или "Caption/Regular" (14/20) [Привязать]
▼ Roboto / Bold / 24px / LH: 32px — 8 слоёв
  └── Замена гарнитуры: Roboto → Inter [Заменить]
```

**Color:**
```
▼ #3366CC (100%) — 23 использования (fills: 18, strokes: 5)
  ├── 15 — не привязаны к Variable/Style
  │   → Похож на "color/primary/500" (#3366CC) ΔE: 0.0 [Привязать]
  ├── 5 — привязаны к Variable "color/primary/500" ✓
  └── 3 — привязаны к Fill Style "Primary/500" ✓
▼ #3366CB (100%) — 4 использования (fills: 4)
  └── 4 — не привязаны
      → Похож на "color/primary/500" (#3366CC) ΔE: 0.4 [Привязать]
```

**Effects:**
```
▼ Drop Shadow: #00000026, Y:4, Blur:12 — 15 нод
  ├── 10 — не привязаны к Effect Style
  │   → Похож на "Shadow/Medium" [Привязать]
  └── 5 — привязаны ✓
```

#### FR-08.3: Массовые операции

- Чекбоксы у каждой группы и каждого элемента

```typescript
// === Общие типы ===

type ScanScope = 'selection' | 'page' | 'document';
type ScanMode = 'typography' | 'colors' | 'effects';

interface ScanOptions {
  scope: ScanScope;
  mode: ScanMode;
  includeInstances: boolean;       // По умолчанию: false
  includeHidden: boolean;          // По умолчанию: true (включать в результаты)
}

interface ScanProgress {
  processed: number;
  total: number;
  phase: 'collecting' | 'analyzing' | 'matching' | 'done' | 'cancelled';
}

// === Результат сканирования: общая обёртка ===

interface ScanNodeMeta {
  nodeId: string;
  nodeName: string;
  nodePath: string;               // "Page / Frame / Card / Text"
  nodeType: string;               // "TEXT", "RECTANGLE", "FRAME", ...
  isHidden: boolean;
  isInsideInstance: boolean;
  instancePath?: string;          // Путь инстанса (если внутри)
  pageId?: string;                // ID страницы (для scope Document)
  pageName?: string;
}

// === Typography Mode ===

interface TypographyGroup {
  key: string;                    // "Inter:Regular:16:24" — уникальный ключ группы
  fontFamily: string;
  fontStyle: string;
  fontSize: number;
  lineHeight: LineHeight;         // { value, unit } | 'AUTO'
  letterSpacing: LetterSpacing;
  nodes: ScanNodeMeta[];
  boundToStyle: boolean;          // Привязан ли к TextStyle
  styleName?: string;             // Имя привязанного стиля
  styleId?: string;
  nearestStyle?: {                // Ближайший существующий стиль
    name: string;
    id: string;
    distance: number;             // 0 = точное совпадение
    diff: string;                 // "fontSize: 15 → 16, lineHeight: 22 → 24"
  };
}

interface FontFamilyGroup {
  family: string;
  styles: { style: string; count: number }[];
  totalCount: number;
  nodes: ScanNodeMeta[];
}

// === Color Mode ===

interface ColorGroup {
  key: string;                    // "#3366CC:1.0:fill" — HEX:opacity:context
  hex: string;
  opacity: number;
  context: 'fill' | 'stroke';
  paintType: string;              // "SOLID", "GRADIENT_LINEAR", ...
  nodes: ScanNodeMeta[];
  boundToVariable: boolean;
  variableName?: string;
  variableId?: string;
  boundToStyle: boolean;
  styleName?: string;
  styleId?: string;
  nearestMatch?: {
    type: 'variable' | 'style';
    name: string;
    id: string;
    deltaE: number;
  };
}

// === Effects Mode ===

interface EffectGroup {
  key: string;                    // "DROP_SHADOW:#00000026:0:4:12:0"
  effectType: string;
  color?: string;                 // HEX
  offset?: { x: number; y: number };
  blur: number;
  spread?: number;
  nodes: ScanNodeMeta[];
  boundToStyle: boolean;
  styleName?: string;
  styleId?: string;
  nearestStyle?: {
    name: string;
    id: string;
    distance: number;
  };
}

// === Оверрайды в инстансах ===

interface InstanceOverride {
  instanceNodeId: string;         // ID ноды-инстанса
  instanceName: string;
  masterComponentName: string;
  overriddenNodeId: string;       // ID дочерней ноды с оверрайдом
  overriddenNodeName: string;
  property: string;               // "fills", "fontFamily", "effectStyleId", ...
  currentValue: any;
  masterValue: any;
  category: 'text-style' | 'color' | 'effect' | 'image';
}

// === Сообщения postMessage ===

// Main → UI
type MainToUIMessage =
  | { type: 'scan-progress'; data: ScanProgress }
  | { type: 'scan-results-typography'; data: { fonts: FontFamilyGroup[]; styles: TypographyGroup[] } }
  | { type: 'scan-results-colors'; data: { groups: ColorGroup[] } }
  | { type: 'scan-results-effects'; data: { groups: EffectGroup[] } }
  | { type: 'scan-results-overrides'; data: { overrides: InstanceOverride[]; images: InstanceOverride[] } }
  | { type: 'action-complete'; data: { affected: number; errors: string[] } }
  | { type: 'available-styles'; data: { textStyles: any[]; fillStyles: any[]; effectStyles: any[] } }
  | { type: 'available-variables'; data: { colorVars: any[]; floatVars: any[] } };

// UI → Main
type UIToMainMessage =
  | { type: 'start-scan'; options: ScanOptions }
  | { type: 'cancel-scan' }
  | { type: 'replace-font'; from: { family: string; style: string }; to: { family: string; style: string }; nodeIds: string[] }
  | { type: 'bind-text-style'; styleId: string; nodeIds: string[] }
  | { type: 'replace-text-value'; field: string; value: any; nodeIds: string[] }
  | { type: 'bind-color-variable'; variableId: string; nodeIds: string[]; context: 'fill' | 'stroke' }
  | { type: 'bind-color-style'; styleId: string; nodeIds: string[]; context: 'fill' | 'stroke' }
  | { type: 'replace-color'; fromHex: string; toHex: string; nodeIds: string[] }
  | { type: 'bind-effect-style'; styleId: string; nodeIds: string[] }
  | { type: 'reset-overrides'; overrideIds: string[] }
  | { type: 'ignore-node'; nodeId: string }
  | { type: 'export-report'; format: 'json' | 'markdown' }
  | { type: 'navigate-to-node'; nodeId: string; pageId?: string }
  | { type: 'close' };
```

### 5.3. Алгоритм сканирования (Orchestrator)

```
[Пользователь нажал "Scan"]
        │
        ▼
[Определить scope] ──► Selection: figma.currentPage.selection
        │               Page: figma.currentPage.children
        │               Document: figma.root.children → loadAsync() each
        ▼
[Собрать все ноды в плоский массив]
        │
        ▼
[Фильтрация]
  ├── Пропустить ноды с pluginData "lint-ignore"
  ├── Пометить isInsideInstance (если флаг includeInstances === false — пропустить)
  └── Пометить isHidden
        │
        ▼
[Чанковая обработка: CHUNK_SIZE = 50]
  ├── Для каждого чанка:
  │   ├── Обработать ноды текущего чанка (в зависимости от mode)
  │   ├── Отправить промежуточные результаты в UI
  │   └── yield → setTimeout(0)
  │
  ▼
[Финализация]
  ├── Суммаризация: группировка по значениям
  ├── Matching: поиск ближайших стилей/переменных
  └── Отправка финальных результатов в UI
```

### 5.4. Обработка оверрайдов инстансов

```
[Сканирование оверрайдов] — отдельный проход (или объединённый)
        │
        ▼
[Для каждого INSTANCE в scope]
  ├── getMainComponentAsync() → masterComponent
  ├── Рекурсивно пройти дочерние ноды инстанса
  ├── Для каждой дочерней ноды:
  │   ├── Найти соответствующую ноду в мастере (по имени + позиции в дереве)
  │   ├── Сравнить стилевые свойства:
  │   │   ├── fills (если type !== IMAGE) → color override?
  │   │   ├── strokes → color override?
  │   │   ├── fillStyleId → style override?
  │   │   ├── fontName, fontSize, lineHeight → text style override?
  │   │   ├── textStyleId → text style override?
  │   │   ├── effects → effect override?
  │   │   └── effectStyleId → effect style override?
  │   ├── Если fills.type === IMAGE и отличается от мастера → в группу "Изображения"
  │   └── Если стилевое отличие → в группу "Оверрайды"
  │
  ▼
[Результат]
  ├── overrides: InstanceOverride[] — стилевые оверрайды
  └── images: InstanceOverride[] — изменённые изображения (отдельная группа)
```

## 7. Технические детали

### 7.1. Color Distance — алгоритм сопоставления цветов

Для поиска ближайшего цвета используется Delta E (CIE76) — Euclidean distance в Lab color space. Упрощённая версия для performance:

```typescript
// Быстрый Euclidean distance в RGB (0-1)
// Достаточно для порога ΔE ≤ 3 при типичных UI-цветах
function colorDistanceRGB(a: RGB, b: RGB): number {
  const dr = (a.r - b.r) * 255;
  const dg = (a.g - b.g) * 255;
  const db = (a.b - b.b) * 255;
  return Math.sqrt(dr * dr + dg * dg + db * db);
}

// Порог по умолчанию: ΔE ≤ 5 (RGB Euclidean)
// 0 = точное совпадение
// ≤5 = почти неотличим визуально
// ≤15 = похожий цвет
const DEFAULT_COLOR_THRESHOLD = 5;
```

### 7.2. Style Matching — сопоставление текстовых стилей

```typescript
interface TextStyleSignature {
  fontFamily: string;
  fontStyle: string;
  fontSize: number;
  lineHeight: number | 'AUTO';
  letterSpacing: number;
}

// Расстояние между двумя текстовыми сигнатурами
function textStyleDistance(a: TextStyleSignature, b: TextStyleSignature): number {
  if (a.fontFamily !== b.fontFamily) return Infinity; // Разные шрифты — не совпадают
  if (a.fontStyle !== b.fontStyle) return 100; // Разные начертания — штраф

  let dist = 0;
  dist += Math.abs(a.fontSize - b.fontSize) * 10;      // fontSize ±1 = 10
  if (a.lineHeight !== 'AUTO' && b.lineHeight !== 'AUTO') {
    dist += Math.abs(a.lineHeight - b.lineHeight) * 5;  // lineHeight ±1 = 5
  }
  dist += Math.abs(a.letterSpacing - b.letterSpacing) * 2;

  return dist;
}

// Найти ближайший TextStyle
async function findNearestTextStyle(
  signature: TextStyleSignature
): Promise<{ style: TextStyle; distance: number; diff: string } | null> {
  const styles = await figma.getLocalTextStylesAsync();
  // ... также проверить библиотечные стили через teamLibrary
  let best: { style: TextStyle; distance: number } | null = null;

  for (const style of styles) {
    const styleSignature = extractTextStyleSignature(style);
    const dist = textStyleDistance(signature, styleSignature);
    if (dist < (best?.distance ?? Infinity)) {
      best = { style, distance: dist };
    }
  }

  if (best && best.distance < 50) { // Порог «похожести»
    return {
      style: best.style,
      distance: best.distance,
      diff: generateDiffString(signature, extractTextStyleSignature(best.style))
    };
  }

  return null;
}
```

### 7.3. Безопасная замена шрифтов (с учётом mixed fonts)

```typescript
async function replaceFont(
  node: TextNode,
  fromFamily: string,
  toFamily: string,
  styleMapping: Map<string, string> // "Regular" → "Regular", "Bold" → "700"
): Promise<void> {
  // Шаг 1: Определить все сегменты с шрифтами
  const segments = node.getStyledTextSegments(['fontName']);

  // Шаг 2: Загрузить все целевые шрифты
  for (const seg of segments) {
    const fn = seg.fontName as FontName;
    if (fn.family === fromFamily) {
      const targetStyle = styleMapping.get(fn.style) || fn.style;
      await figma.loadFontAsync({ family: toFamily, style: targetStyle });
    }
    // Загрузить и исходные шрифты (нужны для доступа к node.characters)
    await figma.loadFontAsync(fn);
  }

  // Шаг 3: Замена по сегментам (с конца, чтобы не сбить индексы)
  // Для setRangeFontName — идём от последнего сегмента к первому
  const sortedSegments = [...segments]
    .filter(s => (s.fontName as FontName).family === fromFamily)
    .sort((a, b) => b.start - a.start);

  for (const seg of sortedSegments) {
    const fn = seg.fontName as FontName;
    const targetStyle = styleMapping.get(fn.style) || fn.style;
    node.setRangeFontName(seg.start, seg.end, {
      family: toFamily,
      style: targetStyle,
    });
  }
}
```

### 7.4. Безопасный сброс оверрайдов инстанса

```typescript
async function resetStyleOverrides(
  instanceNode: InstanceNode,
  overridesToReset: InstanceOverride[]
): Promise<{ success: number; errors: string[] }> {
  const mainComponent = await instanceNode.getMainComponentAsync();
  if (!mainComponent) {
    return { success: 0, errors: ['Мастер-компонент не найден'] };
  }

  let success = 0;
  const errors: string[] = [];

  for (const override of overridesToReset) {
    try {
      const targetNode = await figma.getNodeByIdAsync(override.overriddenNodeId);
      if (!targetNode) {
        errors.push(`Нода ${override.overriddenNodeName} не найдена`);
        continue;
      }

      // Восстановить значение из мастера
      switch (override.property) {
        case 'fills':
          if ('fills' in targetNode) {
            // Копируем fills из соответствующей ноды мастера
            const masterNode = findCorrespondingNode(mainComponent, override.overriddenNodeName);
            if (masterNode && 'fills' in masterNode) {
              (targetNode as GeometryMixin).fills = masterNode.fills;
            }
          }
          break;
        case 'fillStyleId':
          if ('fillStyleId' in targetNode) {
            const masterNode = findCorrespondingNode(mainComponent, override.overriddenNodeName);
            if (masterNode && 'fillStyleId' in masterNode) {
              await (targetNode as any).setFillStyleIdAsync(masterNode.fillStyleId);
            }
          }
          break;
        // ... аналогично для strokes, effects, text properties
      }
      success++;
    } catch (err: any) {
      errors.push(`${override.overriddenNodeName}: ${err.message}`);
    }
  }

  return { success, errors };
}
```

### 7.5. Сопоставление нод инстанса с мастером

```typescript
// Найти соответствующую ноду в мастер-компоненте
// по пути в дереве (имя + позиция)
function findCorrespondingNode(
  masterComponent: ComponentNode,
  targetNodeName: string,
  targetPath?: string[]
): SceneNode | null {
  // Стратегия 1: Поиск по имени (для уникальных имён)
  const byName = masterComponent.findOne(n => n.name === targetNodeName);
  if (byName) return byName;

  // Стратегия 2: Поиск по пути в дереве (для неуникальных имён)
  if (targetPath && targetPath.length > 0) {
    let current: SceneNode | null = masterComponent;
    for (const pathSegment of targetPath) {
      if (current && 'children' in current) {
        current = (current.children as SceneNode[])
          .find(c => c.name === pathSegment) || null;
      } else {
        return null;
      }
    }
    return current;
  }

  return null;
}
```

## 9. Подводные камни и решения

### 9.1. `figma.mixed` — повсюду

Многие свойства нод возвращают `figma.mixed` когда у ноды неоднородные значения (например, TextNode с разными шрифтами). Нужно проверять **каждое свойство** перед использованием:

```typescript
if (node.type === 'TEXT') {
  if (node.fontSize === figma.mixed) {
    // Разбить по сегментам
    const segments = node.getStyledTextSegments(['fontSize']);
  } else {
    // Единое значение
  }
}
```

### 9.2. Immutable массивы (fills, strokes, effects)

Figma возвращает **копии** массивов. Мутация элемента ничего не делает. Всегда: копия → изменение → присвоение.

```typescript
// ❌ Не работает
node.fills[0] = newFill;

// ✅ Работает
const fills = [...node.fills];
fills[0] = newFill;
node.fills = fills;
```

### 9.3. Шрифты должны быть загружены

Перед любой модификацией текста — обязательный `loadFontAsync()`. Если забыть — runtime error.

### 9.4. `findAll()` на весь документ — медленно

Не использовать `figma.root.findAll()` или `page.findAll()`. Вместо этого — ручной рекурсивный обход с чанкованием.

### 9.5. Скрытые ноды — API полностью рабочий

Ноды с `visible: false` полностью доступны для чтения и модификации через API. Figma не блокирует работу со скрытыми слоями. Проблема только в UX — пользователь не ожидает изменений скрытых элементов.

### 9.6. Оверрайды инстансов — нет прямого API

Figma не предоставляет API `getOverrides()`. Для определения оверрайдов нужно сравнивать свойства дочерних нод инстанса с соответствующими нодами мастер-компонента вручную. Это ресурсоёмкая операция из-за необходимости `getMainComponentAsync()` и рекурсивного обхода обоих деревьев.

### 9.7. `getMainComponentAsync()` может вернуть `null`

Если мастер-компонент находится в другом файле (библиотека) и ещё не загружен — может вернуть `null`. Нужна обработка этого случая.

### 9.8. Race conditions при Document scope

При сканировании всего документа нужно последовательно загружать страницы через `loadAsync()`. Параллельная загрузка может вызвать сбои.
