# Сравнительный анализ архитектуры и стратегий именования дизайн-токенов в ведущих индустриальных дизайн-системах

Дизайн-токены представляют собой не просто переменные в коде, а фундаментальную методологию систематизации визуальных решений, обеспечивающую синхронизацию между дизайном и разработкой на различных технологических платформах. В современной индустрии дизайн-системы перешли от жесткого кодирования значений к сложным многоуровневым структурам именования, которые позволяют масштабировать продукты, поддерживать динамические темы и автоматизировать передачу макетов в производство. Данный отчет представляет собой глубокое исследование конвенций именования (naming conventions) в восьми ключевых дизайн-системах, анализируя их синтаксис, иерархию и семантическую нагрузку, а также формулирует рекомендации для построения новых open-source решений.

## 1. Material Design 3 (Google): Иерархия системных ролей и динамическая адаптивность

Material Design 3 (M3) устанавливает один из наиболее строгих и научно обоснованных подходов к именованию токенов, что обусловлено необходимостью поддержки технологии динамического цвета (Dynamic Color). Система ориентирована на абстрагирование визуальных свойств от их физических значений, что позволяет интерфейсу адаптироваться к пользовательским предпочтениям в реальном времени.

### Формат именования и синтаксическая структура

В системе Material Design 3 используется точечная нотация (dot notation) в качестве основного разделителя уровней иерархии: `md.sys.color.primary`. Такой подход позволяет логически группировать токены и упрощает их поиск в JSON-структурах и программных объектах. В веб-реализации эти токены часто трансформируются в CSS-переменные с использованием дефисов в качестве разделителей и специфических префиксов, таких как `--md-sys-` или `--mat-sys-` в Angular Material. Регистр сегментов обычно придерживается kebab-case для составных слов (например, `on-primary-container`), что обеспечивает максимальную совместимость с веб-стандартами.

### Классификация и уровни абстракции

M3 выделяет три четко разграниченных класса токенов, которые формируют иерархическую цепочку зависимостей. Понимание этой структуры критически важно для разработчиков, стремящихся к созданию гибких тем.

1. **Reference Tokens (ref):** Представляют собой полный спектр доступных опций в системе. Они указывают на статические значения, такие как конкретные hex-коды палитры или размеры в dp. Пример: `md.ref.palette.primary40` или `md.ref.typeface.brand-regular`.
    
2. **System Tokens (sys):** Семантический слой, определяющий роли токенов внутри интерфейса. Эти токены не имеют фиксированного значения, а ссылаются на референсные токены в зависимости от контекста (светлая или темная тема). Пример для цвета: `md.sys.color.surface-variant`; для типографики: `md.sys.typescale.label-large`.
    
3. **Component Tokens (comp):** Самый специфичный уровень, закрепляющий системные или референсные токены за конкретными элементами компонентов. Пример: `md.comp.filled-button.container.color` или `md.comp.fab.primary.icon.color`.
    

### Паттерны и состояния

Именование в M3 строится на ролевой модели (role-based), где имя токена описывает его функцию (например, `on-primary` — цвет текста на основном фоне), а не конкретный оттенок. Состояния компонентов, такие как `hover`, `focused`, `pressed` и `disabled`, интегрируются в структуру компонентных токенов и группируются в спецификациях. Состояние `enabled` часто выступает как подразумеваемое (default), но в программных интерфейсах оно может указываться явно для ясности иерархии.

## 2. Shopify Polaris: Прагматизм в масштабируемых шкалах

Дизайн-система Polaris ориентирована на экосистему электронной коммерции, где критически важна предсказуемость и быстрота разработки сторонними участниками. Конвенции Polaris подчеркивают использование человекочитаемого языка и масштабируемых числовых шкал.

### Синтаксис и форматирование

Polaris использует дефис (`-`) в качестве основного разделителя для CSS-переменных, что является стандартом для веба: `--p-color-text-interactive-hover`. При этом в среде JavaScript токены преобразуются в lower camelCase (например, `colorBlueLighter`), обеспечивая естественный интерфейс для разработчиков на React. Префикс `--p-` служит обязательным пространством имен (namespace) для предотвращения конфликтов со стилями сторонних приложений Shopify.

### Иерархическая структура и примеры

Система опирается на формулу: `группа + подгруппа + вариант`. Такой подход позволяет легко расширять систему без изменения существующей логики.

|**Уровень (Tier)**|**Пример (Цвет)**|**Пример (Размер/Шрифт)**|**Логика именования**|
|---|---|---|---|
|**Primitive**|`--p-gray-100`|`--p-space-4`|Прямое значение или шкала|
|**Semantic**|`--p-color-text-critical`|`--p-font-size-base`|Функциональное назначение|
|**Component**|`--p-button-bg-hover`|`--p-input-border-error`|Привязка к конкретному элементу|

### Модификаторы и логика состояний

В Polaris состояния (hover, active, focus) всегда располагаются в конце имени токена: `--p-color-text-interactive-hover`. Для числовых шкал система использует принцип «от центра к краям», где `base` является значением по умолчанию, а приставки `small-` и `large-` с числовыми индексами обозначают уменьшение или увеличение относительно базы: `small-100`, `base`, `large-100`. Такое решение делает систему интуитивно понятной при выборе размеров отступов и шрифтов.

## 3. IBM Carbon: Многослойная модель и контекстуальная зависимость

IBM Carbon выделяется среди конкурентов своей уникальной моделью «слоев» (layering), которая решает проблемы визуальной иерархии в сложных корпоративных дашбордах. Именование токенов в Carbon напрямую отражает эту архитектурную особенность.

### Формат и специфические обозначения

В системе Carbon разделителем уровней служит дефис, а для обозначения наборов слоев используются числовые суффиксы: `-00`, `-01`, `-02`, `-03`. В контексте Sass используются переменные со знаком доллара (например, `$layer-01`), а в CSS — стандартные переменные `--cds-`. Регистр всегда остается строгим kebab-case.

### Структура имени и паттерн CTI

Carbon активно использует паттерн, близкий к CTI (Category/Type/Item), но адаптированный под роли интерфейса.

- **Primitive (Global):** Определяют базовую палитру IBM Design Language, например `blue-60` или `gray-10`.
    
- **Semantic (Alias):** Токены, описывающие роль элемента в системе. Здесь проявляется модель слоев: `$background` (база), `$layer-01` (первый уровень вложенности), `$field-02` (поле ввода на первом слое).
    
- **Component:** Токены для конкретных частей компонентов, например `$button-primary-active` или `$text-on-color`.
    

### Контекстуальные токены и состояния

Одной из самых инновационных черт Carbon является использование «контекстуальных токенов». Это токены, имена которых не содержат числового суффикса (например, `$layer`, `$field`). В коде их значение меняется автоматически в зависимости от того, в какой CSS-класс или компонент `Layer` они обернуты. Это позволяет дизайнерам и разработчикам создавать один вариант компонента, который будет выглядеть корректно на любом уровне вложенности. Состояния в Carbon всегда указываются в конце имени: `$background-hover`, `$button-danger-active`. Состояние `default` считается имплицитным и не прописывается в названии.

## 4. Atlassian Design System: Функциональная иерархия и семантическая интенсивность

Система Atlassian строится на принципах ясности намерений и предотвращения ошибок при выборе цвета или отступа. Конвенции именования здесь призваны обеспечить максимальную доступность (accessibility) и удобство темизации.

### Формат и синтаксис

Atlassian использует точечную нотацию в документации и инструментах дизайна (`color.text.subtle`), которая при компиляции в CSS превращается в дефисную нотацию с префиксом `--ds-`: `--ds-text-subtle`. Разработчики могут использовать функцию `token()`, которая принимает имя токена через точку, обеспечивая проверку типов в JavaScript и TypeScript.

### Структура и паттерн именования

Паттерн Atlassian следует логике `Foundation.Property.Modifier`.

- **Primitive:** Примитивные значения (например, `Blue700`) в Atlassian часто скрыты от прямого использования разработчиками приложений, чтобы гарантировать использование семантических имен, поддерживающих темную тему.
    
- **Semantic:** Основные токены, описывающие роль. Примеры: `color.background.danger`, `color.text.success`, `color.border.input`.
    
- **Component:** Применяются для специфических элементов, когда семантических токенов недостаточно. Пример: `color.background.button.primary.hovered`.
    

### Модификаторы интенсивности и состояний

Atlassian использует уникальные модификаторы интенсивности, такие как `subtle`, `subtlest`, `bold` и `bolder`. Это позволяет создавать иерархию внутри одного функционального типа (например, несколько уровней «второстепенного» текста). Состояния всегда добавляются в конец имени и используют форму прошедшего времени для глаголов: `hovered`, `pressed`, `selected`, `focused`, `disabled`. Слово `default` иногда используется явно в новых наборах токенов (например, `success.subtler.default`), но чаще оно опускается, если роль токена понятна из контекста.

## 5. Salesforce Lightning Design System (SLDS): Монументальная архитектура Styling Hooks

Salesforce была одной из первых компаний, внедривших концепцию дизайн-токенов в 2014 году. Современная версия системы (SLDS 2) перешла к использованию Styling Hooks — стандарта CSS-переменных с крайне детализированной структурой.

### Формат и синтаксис Styling Hooks

SLDS использует дефис в качестве единственного разделителя и строгий регистр kebab-case для CSS-переменных. Однако в старых версиях системы (SLDS 1) широко применялся camelCase для имен токенов, что до сих пор встречается в коде LWC (Lightning Web Components): `--lwc-spacingSmall`. В новой архитектуре используется префикс `--slds-g-` для глобальных токенов и `--slds-c-` для компонентных.

### Детальный паттерн именования

Таксономия Salesforce является самой длинной и подробной среди всех изученных систем. Она включает до девяти сегментов: `--[namespace]-[scope]-[category]-[property]-[pairing]-[role]-[attribute]-[state]-[range]`.

|**Сегмент**|**Описание**|**Пример**|
|---|---|---|
|**Namespace**|Система, владеющая хуком|`slds`|
|**Scope**|Область действия (Global/Component)|`g` или `c`|
|**Category**|Категория дизайна|`color`, `spacing`|
|**Property**|Свойство UI|`border`, `background`|
|**Role**|Семантическая роль|`brand`, `error`, `accent`|
|**Range**|Числовой индекс шкалы|`1`, `2`, `50`, `60`|

### Примеры по уровням

1. **Primitive:** В SLDS 2 примитивы часто заменяются на «доступные палитры» (accessible palettes): `--slds-g-color-palette-blue-50`.
    
2. **Semantic:** Основные глобальные стили: `--slds-g-color-brand-base-1` или `--slds-g-spacing-4`.
    
3. **Component:** Специфичные для элементов: `--slds-c-button-brand-color-background`.
    

Состояния в SLDS располагаются ближе к концу имени, но перед числовым индексом шкалы, если он присутствует: `--slds-g-color-brand-base-hover-60`. Состояние `default` обычно подразумевается под ключевым словом `base`.

## 6. Adobe Spectrum: Естественный язык и плоская структура

Adobe Spectrum придерживается философии «человекочитаемости» (human-readability), стремясь сделать токены понятными не только инженерам, но и дизайнерам и менеджерам продуктов.

### Философия именования

Spectrum осознанно отказывается от глубоко вложенных иерархических структур в пользу «плоской структуры» (flat structure). Это позволяет именам токенов звучать как естественные фразы. Разделителем служит дефис, регистр — kebab-case.

### Структура Context-Unit-Clarification

Имя токена в Spectrum строится по трехчастной модели: контекст, общая единица (common unit) и уточнение (clarification).

- **Primitive (Global):** Используются названия цветов и числовые индексы: `gray-100`, `blue-400`. Эти токены формируют «сырую» палитру системы.
    
- **Semantic (Alias):** Токены, описывающие использование без привязки к компоненту. Пример: `accent-color-100`.
    
- **Component:** Примеры включают весьма описательные имена: `checkbox-control-size-small` или `action-button-edge-to-hold-icon-large`.
    

### Масштабирование и размеры

Spectrum активно использует систему T-shirt размеров (xsmall, small, medium, large, xlarge) для большинства свойств, таких как отступы и размеры компонентов. Состояния интегрируются в имя органично: `button-primary-background-color-hover`. Дефолтный размер (`medium`) часто считается подразумеваемым, но может быть указан явно в спецификациях для полноты картины.

## 7. GitHub Primer: Гибридный синтаксис и ориентация на паттерны

GitHub Primer сочетает в себе строгость функционального подхода с гибкостью, необходимой для поддержки множества различных веб-сервисов компании.

### Формат и специфические разделители

Primer использует уникальный гибридный подход к форматированию сегментов имени. В то время как блоки разделяются дефисами (для CSS) или точками (для JS), сами названия свойств внутри блоков пишутся в camelCase. Например: `base-fontWeight-semibold` или `control-danger-borderColor-rest`. Это позволяет сократить общую длину имени, сохраняя читаемость составных слов.

### Категории и иерархия

Primer разделяет токены на три функциональные группы :

1. **Base:** Самый низкий уровень, мапинг на сырые значения. Пример: `base-size-4`, `base-color-green-5`.
    
2. **Functional:** Представляют глобальные паттерны интерфейса. Пример: `bgColor-inset`, `borderColor-default`.
    
3. **Component/Pattern:** Специфичны для компонентов или групп компонентов (например, `control` для кнопок и инпутов). Пример: `button-primary-bgColor-hover`, `text-codeInline-fontSize`.
    

### Обозначение состояний и Scale

GitHub использует ключевое слово `rest` для обозначения состояния покоя, что является отличительной чертой системы: `fgColor-rest`, `bgColor-rest`. Другие состояния включают `hover`, `active`, `disabled`. Позиция состояния всегда конечная. Для размеров используются как T-shirt размеры, так и термины плотности: `condensed`, `normal`, `spacious`.

## 8. Open Props: Атомарный CSS и минималистичный подход

Open Props — это не классическая дизайн-система в корпоративном понимании, а скорее набор атомарных дизайн-токенов, реализованных как чистые CSS-переменные.

### Формат и структура

Open Props использует максимально простую структуру: `--категория-индекс`. Разделителем служит дефис, регистр — kebab-case. Например: `--size-1`, `--radius-2`, `--indigo-5`.

### Уровни абстракции и использование

1. **Primitive:** Прямые значения свойств. Например, `--font-size- fluid-1` для адаптивной типографики.
    
2. **Semantic (через Aliasing):** Пользователям предлагается создавать свои семантические слои. Система поддерживает внутренние переменные с префиксом подчеркивания для локального контекста компонентов: `--_bg: var(--surface-2)`.
    
3. **Dynamic Palette:** Система динамических палитр позволяет использовать токены типа `var(--color-7)`, которые меняют свои значения на основе базовых переменных темы (`--palette-hue`, `--palette-chroma`).
    

Состояния в Open Props не стандартизированы на уровне имен токенов, так как система ориентирована на нативный CSS, где состояния обрабатываются через селекторы (например, `:hover`), в которых просто переопределяются значения локальных переменных.

Ниже приведены сопоставимые примеры токенов для трех уровней абстракции в каждой системе.

|**Система**|**Primitive (Color/Size)**|**Semantic (Role/Surface)**|**Component (Element/State)**|
|---|---|---|---|
|**Material 3**|`md.ref.palette.blue60`|`md.sys.color.primary`|`md.comp.button.container.color`|
|**Polaris**|`--p-blue-500`|`--p-color-text-success`|`--p-button-bg-primary-hover`|
|**Carbon**|`$blue-60`|`$layer-01`|`$button-primary-active`|

## ТЕМАТИЧЕСКИЙ АНАЛИЗ СОСТОЯНИЙ И МОДИФИКАТОРОВ

Анализ восьми систем выявил несколько ключевых архитектурных тенденций в обработке состояний (states) и модификаторов.

### Позиционирование состояний

Абсолютное большинство современных систем (Material 3, Polaris, Carbon, Atlassian, Spectrum, Primer) помещает состояние в самый конец строки имени токена. Это обусловлено психологией поиска в инструментах разработки: разработчик сначала ищет объект (кнопка), затем его свойство (фон), а в конце уточняет состояние (наведение). Исключением является Salesforce SLDS, где состояние может находиться в середине цепочки перед числовым индексом, что делает структуру более жесткой, но менее интуитивной для быстрого чтения.

### Явное vs. Неявное обозначение Default

Существует два основных подхода к именованию состояния по умолчанию.

1. **Имплицитный (Неявный):** Carbon, Atlassian (частично), Spectrum и MD3 предпочитают опускать слово `default`. Если токен называется `button-primary-bg`, это автоматически означает состояние покоя. Это сокращает длину имен и уменьшает визуальный шум.
    
2. **Эксплицитный (Явный):** GitHub Primer (`rest`) и Shopify Polaris (`base`) вводят специальные ключевые слова для состояния по умолчанию. Это значительно повышает строгость системы и упрощает автоматизированную обработку токенов, так как исключает ситуацию, когда отсутствие сегмента в имени может быть истолковано двояко.
    

### Модификаторы интенсивности

Системы Atlassian и Polaris используют семантические модификаторы вместо числовых там, где это помогает дизайнеру принять решение. Вместо `gray-100/200/300`, Atlassian предлагает `subtlest`, `subtle`, `default`, `bold`. Это третья степень абстракции, которая отдаляет дизайн от физики цвета в сторону интерфейсной логики.

## РЕКОМЕНДАЦИЯ ДЛЯ НОВОЙ OPEN-SOURCE ДИЗАЙН-СИСТЕМЫ

На основе проведенного анализа и современных стандартов (W3C DTCG), предлагается архитектура именования для новой кросс-платформенной системы, ориентированной на Web, iOS и Android.

### 1. Базовый синтаксис и формат (DTCG)

Новая система должна использовать JSON-формат, соответствующий спецификации Design Tokens Community Group (DTCG).

- **Разделитель в источнике:** Точка (`.`). Это естественный стандарт для объектов JSON и навигации в коде Swift/Kotlin.
    
- **Регистр сегментов:** kebab-case. Это обеспечивает лучшую читаемость и совместимость с CSS без необходимости сложной трансформации имен.
    
- **Трансформация:** Имена должны автоматически преобразовываться через Style Dictionary.
    

### 2. Рекомендуемая структура имени (Taxonomy)

Предлагается паттерн из пяти уровней, который обеспечивает баланс между краткостью и описательностью:

`[Система].[Категория].[Объект/Роль].[Свойство].[Состояние/Модификатор]`

- **Пример Primitive:** `os.color.blue.600`, `os.space.stack.400`
    
- **Пример Semantic:** `os.color.text.primary`, `os.color.surface.default`
    
- **Пример Component:** `os.button.primary.background.hover`
    

### 3. Реализация состояний и "Default"

Рекомендуется использовать **явное** обозначение состояния по умолчанию с помощью ключевого слова `default`.

- Преимущество: Это делает структуру предсказуемой для новых пользователей и упрощает написание скриптов трансформации.
    
- Пример: `action.primary.bg.default`, `action.primary.bg.hover`, `action.primary.bg.disabled`.
    

### 4. Кросс-платформенное отображение

Для обеспечения бесшовной работы на всех платформах, токены должны транслироваться следующим образом:

#### Web (CSS)

Использование дефисов и двойного тире.

CSS

```
--os-color-text-primary: #1a1a1a;
--os-button-primary-bg-hover: var(--os-color-blue-700);
```

#### iOS (Swift)

Использование вложенных перечислений (enum) или структур, что позволяет использовать преимущества автодополнения Xcode.

Swift

```
public enum DesignTokens {
    public enum Color {
        public static let textPrimary = UIColor(named: "os-color-text-primary")
    }
    public enum Button {
        public enum Primary {
            public static let backgroundHover = UIColor(named: "os-button-primary-bg-hover")
        }
    }
}
```

#### Android (Kotlin)

Использование `object` для создания иерархического пространства имен, совместимого с Jetpack Compose.

Kotlin

```
object DesignTokens {
    object Color {
        val TextPrimary = Color(0xFF1A1A1A)
    }
    object Button {
        object Primary {
            val BackgroundHover = Color(0xFF2B6CB0)
        }
    }
}
```

### 5. Принципы масштабируемости

- **Пространство имен (Namespace):** Всегда начинать с короткого уникального префикса (например, `os.` для Open System), чтобы избежать коллизий при интеграции в другие проекты.
    
- **Отсутствие сокращений:** Использовать полные слова (`background` вместо `bg`, `color` вместо `clr`). Это снижает порог входа для новых членов команды.
    
- **Числовые шкалы для размеров:** Использовать 100-900 или кратные 100 значения (как в MD3 и Polaris), так как это позволяет легко вставлять промежуточные значения в будущем (например, 450 между 400 и 500) без переименования всей системы.
    

Такой подход гарантирует, что дизайн-система будет не только технически совершенной, но и интуитивно понятной для всех участников процесса, обеспечивая "единый источник истины" (single source of truth) на протяжении всего жизненного цикла продукта.
