# Технический анализ спецификации W3C Design Tokens Community Group (DTCG) версии 2025.10: Стандарт системного проектирования нового поколения

Официальная публикация первой стабильной версии спецификации Design Tokens Community Group (DTCG) 2025.10, состоявшаяся 28 октября 2025 года, ознаменовала собой фундаментальный сдвиг в индустрии создания цифровых продуктов. Этот документ, подготовленный консорциумом экспертов из таких компаний, как Adobe, Amazon, Google, Figma и Microsoft, предлагает универсальный язык для обмена дизайнерскими решениями, который выходит за рамки проприетарных форматов отдельных инструментов. Спецификация 2025.10 не просто фиксирует текущие практики, но и закладывает основу для работы с современными цветовыми пространствами, сложными композитными структурами и многослойными системами темизации, обеспечивая бесшовную интероперабельность между средой проектирования и кодом.

## Глубинный анализ формата файла и архитектурных ограничений

В основе спецификации лежит выбор JSON в качестве формата обмена данными, что обусловлено его универсальностью, поддержкой в стандартных библиотеках практически всех языков программирования и относительной человекочитаемостью. Однако версия 2025.10 вводит строгие семантические правила, превращающие обычный JSON в специализированный формат `application/design-tokens+json`.

### Структура и зарезервированные свойства токена

Каждый объект в файле токенов интерпретируется либо как группа, либо как сам токен. Определяющим фактором является наличие зарезервированного свойства `$value`. Если оно присутствует, объект считается токеном; в противном случае — группой.

|**Свойство**|**Тип данных**|**Обязательность**|**Техническое назначение и контекст**|
|---|---|---|---|
|`$value`|Any|Обязательно для токена|Фактическое значение дизайнерского решения. Тип данных зависит от `$type`.|
|`$type`|String|Рекомендуется|Семантическая классификация (например, `color`, `dimension`). Позволяет инструментам трансляции, таким как Style Dictionary, выбирать правильные алгоритмы преобразования.|
|`$description`|String|Опционально|Метаданные для документации. Используются инструментами вроде zeroheight для автоматической генерации гайдлайнов.|
|`$extensions`|Object|Опционально|Контейнер для проприетарных данных. Позволяет вендорам добавлять специфические функции без нарушения совместимости со стандартом.|

Спецификация строго запрещает смешивание токенов и дочерних групп внутри одного объекта. Объект не может одновременно содержать `$value` и вложенные ключи, представляющие другие токены. Нарушение этого правила приводит к невалидности всего документа, что критично для парсеров, работающих в автоматизированных CI/CD пайплайнах.

### Иерархия и правила именования групп (Group Nesting)

Группировка токенов позволяет организовать иерархию по категориям (например, `brand`, `component`, `semantic`). Спецификация 2025.10 поддерживает бесконечную вложенность групп. Однако для обеспечения надежности ссылок введены жесткие ограничения на символы в именах групп и токенов. Имена не могут содержать точки (`.`), так как они используются в качестве разделителей путей в JSON Pointer и алиасах. Также запрещены фигурные скобки (`{}`) и символы, начинающиеся с `$`, за исключением зарезервированных свойств самой спецификации.

Для случаев, когда группе необходимо присвоить базовое значение (например, цвет фона, который также имеет варианты), введена концепция `$root` токена. Это позволяет избежать конфликта имен, когда группа `surface` должна иметь собственное значение и при этом содержать дочерний токен `surface.variant`.

|**Аспект формата**|**Стандартное значение / Правило**|
|---|---|
|**Media Type**|`application/design-tokens+json`|
|**Расширение файла**|`.tokens.json` (рекомендуется) или `.tokens`|
|**Кодировка**|UTF-8|
|**Регистрозависимость**|Имена ключей регистрозависимы|

## Детальный разбор типов токенов ($type) и их спецификаций

Спецификация 2025.10 определяет 13 фундаментальных типов, разделенных на атомарные и композитные. Типизация является ключевым механизмом, позволяющим инструментам трансляции понимать, как преобразовывать абстрактное значение в платформенно-зависимый код (например, из `dimension` в `rem` для веба или в `dp` для Android).

### Атомарные типы данных

Атомарные типы представляют собой элементарные единицы стиля, которые не могут быть разделены далее без потери смысла.

#### 1. Color (Цвет)

Самый сложный и динамично развивающийся тип. В версии 2025.10 он перестал быть просто строкой с HEX-кодом и превратился в объект, поддерживающий 14 цветовых пространств.

JSON

```
{
  "action-primary": {
    "$type": "color",
    "$value": {
      "colorSpace": "oklch",
      "components": [0.65, 0.25, 260],
      "alpha": 1,
      "hex": "#3b82f6"
    }
  }
}
```

- **Допустимые значения**: Объект с `colorSpace` и массивом `components`.
    
- **Дизайн-система**: Используется для определения палитр, состояний интерактивных элементов и брендовых цветов.
    

#### 2. Dimension (Размерность)

Описывает любые физические размеры. Значение всегда является строкой, объединяющей число и единицу измерения.

JSON

```
{
  "spacing-layout-m": {
    "$type": "dimension",
    "$value": "24px"
  }
}
```

- **Допустимые значения**: Строка вида `[number][unit]`.
    
- **Дизайн-система**: Применяется для отступов (margin, padding), радиусов скругления, ширины контейнеров.
    

#### 3. Font Family (Семейство шрифтов)

Представляет собой упорядоченный список имен шрифтов.

JSON

```
{
  "font-interface": {
    "$type": "fontFamily",
    "$value": ["Inter Variable", "Inter", "system-ui", "sans-serif"]
  }
}
```

- **Допустимые значения**: Массив строк.
    
- **Дизайн-система**: Определение основных и вспомогательных гарнитур интерфейса.
    

#### 4. Font Weight (Насыщенность шрифта)

Может быть выражена числом или стандартным ключевым словом.

|**Ключевое слово**|**Числовое значение**|
|---|---|
|`thin`, `hairline`|100|
|`regular`, `normal`|400|
|`bold`|700|
|`black`, `heavy`|900|

JSON

```
{
  "font-weight-accent": {
    "$type": "fontWeight",
    "$value": 600
  }
}
```

#### 5. Duration (Длительность)

Временной интервал для анимационных процессов.

JSON

```
{
  "motion-duration-smooth": {
    "$type": "duration",
    "$value": "300ms"
  }
}
```

- **Допустимые значения**: Строка с единицами `ms` или `s`.
    

#### 6. Cubic Bézier (Кривая Безье)

Математическое описание функции сглаживания анимации.

JSON

```
{
  "motion-easing-standard": {
    "$type": "cubicBezier",
    "$value": [0.4, 0, 0.2, 1]
  }
}
```

- **Допустимые значения**: Массив из четырех чисел в диапазоне от 0 до 1 для X-координат.
    

#### 7. Number (Число)

Безразмерная числовая величина.

JSON

```
{
  "line-height-multiplier": {
    "$type": "number",
    "$value": 1.5
  }
}
```

- **Дизайн-система**: Коэффициенты масштабирования, прозрачность (если не в составе цвета), множители межстрочных интервалов.
    

#### 8. Stroke Style (Стиль линии)

Описывает тип начертания границ или линий.

JSON

```
{
  "border-style-focus": {
    "$type": "strokeStyle",
    "$value": "dashed"
  }
}
```

- **Допустимые значения**: `solid`, `dashed`, `dotted` или сложный объект для штриховки.
    

### Композитные типы данных (Composite Tokens)

Композитные токены — это высокоуровневые абстракции, которые объединяют несколько атомарных типов в единую логическую структуру. Они критически важны для описания сложных стилей компонентов, таких как карточки, кнопки или текстовые блоки.

#### 9. Typography (Типографика)

Стиль оформления текста, объединяющий шрифтовые параметры.

JSON

```
{
  "text-body-m": {
    "$type": "typography",
    "$value": {
      "fontFamily": "{font.interface}",
      "fontSize": "16px",
      "fontWeight": "{font.weight.normal}",
      "lineHeight": 1.5,
      "letterSpacing": "0px"
    }
  }
}
```

#### 10. Shadow (Тень)

Поддерживает многослойные эффекты через массив объектов.

JSON

```
{
  "elevation-l": {
    "$type": "shadow",
    "$value":, "alpha": 0.15 },
        "offsetX": "0px",
        "offsetY": "8px",
        "blur": "16px",
        "spread": "0px"
      },
      {
        "color": "{color.shadow.ambient}",
        "offsetX": "0px",
        "offsetY": "2px",
        "blur": "4px",
        "spread": "0px"
      }
    ]
  }
}
```

#### 11. Border (Граница)

Комплексное описание обводки элемента.

JSON

```
{
  "border-interactive": {
    "$type": "border",
    "$value": {
      "color": "{color.action.primary}",
      "width": "2px",
      "style": "solid"
    }
  }
}
```

#### 12. Transition (Переход)

Связывает все параметры анимации перехода в один токен.

JSON

```
{
  "transition-fade": {
    "$type": "transition",
    "$value": {
      "duration": "{motion.duration.fast}",
      "delay": "0ms",
      "timingFunction": "{motion.easing.standard}"
    }
  }
}
```

#### 13. Gradient (Градиент)

Список цветовых переходов с указанием позиций.

JSON

```
{
  "surface-gradient-brand": {
    "$type": "gradient",
    "$value": [
      { "color": "{color.brand.primary}", "position": 0 },
      { "color": "{color.brand.secondary}", "position": 1 }
    ]
  }
}
```

## Механизмы алиасов и разрешение ссылок (Resolver Module)

Алиасы — это "кровеносная система" дизайн-токенов, позволяющая строить многоуровневые системы, где семантические токены (например, `button-bg`) ссылаются на примитивные значения (например, `blue-500`).

### Синтаксис ссылок и JSON Pointer

Спецификация 2025.10 поддерживает два ключевых метода ссылок:

1. **Стандартный синтаксис `{path.to.token}`**: Предназначен для ссылок на токен целиком. Инструменты обязаны разрешать такие ссылки до получения конечного значения.
    
2. **JSON Pointer (RFC 6901)**: Использует свойство `$ref` для точечного обращения к свойствам внутри композитных токенов или массивов. Это позволяет, например, взять только цвет из токена границы.
    

Пример использования JSON Pointer для извлечения отдельного свойства:

JSON

```
{
  "custom-token": {
    "$value": { "$ref": "#/border.interactive/$value/color" }
  }
}
```

### Правила разрешения (Resolution)

Процесс разрешения зависимостей в спецификации 2025.10 — это строгий четырехэтапный алгоритм, гарантирующий детерминированность результата :

- **Валидация (Input Validation)**: Проверка корректности всех входящих данных и соответствие схеме.
    
- **Упорядочивание (Ordering)**: Создание плоской структуры из нескольких файлов или тем согласно массиву `resolutionOrder`. Здесь работает правило "последний в списке перекрывает предыдущих".
    
- **Разрешение алиасов (Aliases)**: На этом этапе все строковые ссылки заменяются на фактические значения. Спецификация поддерживает глубокие вложенные ссылки.
    
- **Финальное разрешение (Resolution)**: Формирование итогового набора токенов для потребления инструментами трансляции.
    

**Циклические ссылки**: Любая попытка создать циклическую зависимость (Токен А -> Токен Б -> Токен А) должна приводить к немедленной ошибке парсинга. Инструменты обязаны реализовывать алгоритмы обнаружения циклов для предотвращения бесконечной рекурсии.

**Многофайловая поддержка**: Спецификация 2025.10 официально вводит механизм объединения файлов черезResolver. В корневом документе-резолвере описывается массив `sources`, который может содержать как локальные относительные пути, так и внешние URL-адреса. Это позволяет командам разделять базовые палитры и специфические темы (светлая/темная) по разным файлам, сохраняя единую логику сборки.

## Современные цветовые пространства в DTCG 2025.10

Переход от sRGB к современным цветовым моделям — одно из самых значимых достижений версии 2025.10. Современные дисплеи (OLED, Liquid Retina) способны отображать цвета за пределами стандартного охвата sRGB, и спецификация предоставляет инструменты для их описания.

### Поддержка 14 цветовых пространств

Спецификация включает в себя полную поддержку следующих пространств :

1. `srgb` — стандарт для старых мониторов и веба.
    
2. `display-p3` — расширенный охват для современных устройств.
    
3. `oklch` — перцептивно равномерное пространство для генерации палитр.
    
4. `oklab` — прямоугольные координаты для OKLCH.
    
5. `lab` — стандарт CIE для точного измерения цвета.
    
6. `lch` — цилиндрическая форма Lab.
    
7. `srgb-linear` — используется для расчетов освещения в 3D.
    
8. `a98-rgb`, `prophoto-rgb`, `rec2020` — профессиональные фото и видео стандарты.
    
9. `xyz-d65`, `xyz-d50`, `hsl`, `hwb`.
    

### Технический синтаксис и примеры

Для каждого пространства определен свой набор компонентов в массиве `components`.

#### OKLCH (Оптимальный выбор для дизайн-систем)

Это пространство позволяет изменять оттенок (Hue), сохраняя при этом ту же визуальную яркость (Lightness). Это критично для создания доступных (accessible) интерфейсов.

JSON

```
{
  "brand-vibrant": {
    "$type": "color",
    "$value": {
      "colorSpace": "oklch",
      "components": [0.62, 0.28, 256.4],
      "alpha": 1
    }
  }
}
```

- **Рекомендация**: Используйте OKLCH для автоматической генерации темных и светлых тем, так как шкала Lightness (L) здесь предсказуема в отличие от HSL.
    

#### Display P3

Необходим для передачи сверхнасыщенных цветов, которые физически невозможно отобразить в sRGB.

JSON

```
{
  "neon-green": {
    "$type": "color",
    "$value": {
      "colorSpace": "display-p3",
      "components": [0.46, 0.98, 0.12]
    }
  }
}
```

- **Рекомендация**: Применяйте P3 для акцентных элементов на мобильных устройствах, но всегда предоставляйте `hex` fallback для обратной совместимости.
    

#### CIELAB (Lab)

Используется для научного определения цвета, независимого от устройства.

JSON

```
{
  "accurate-gray": {
    "$type": "color",
    "$value": {
      "colorSpace": "lab",
      "components": 
    }
  }
}
```

## Расширения ($extensions) и мета-информация

Свойство `$extensions` служит механизмом "безопасного расширения" стандарта. Оно позволяет командам и разработчикам инструментов хранить данные, которые еще не стали частью официальной спецификации, но необходимы для рабочих процессов.

### Назначение и синтаксис

Данные внутри `$extensions` должны быть организованы с использованием уникальных ключей вендоров (обычно это обратная доменная нотация, например `com.figma.plugins`). Это предотвращает конфликты имен между различными инструментами, обрабатывающими один и тот же файл.

### Примеры использования для темизации и платформ

Расширения часто используются для определения того, как токен должен вести себя в разных "режимах" или на разных платформах, если это не вынесено в отдельные файлы Resolver.

JSON

```
{
  "surface-default": {
    "$type": "color",
    "$value": "#ffffff",
    "$extensions": {
      "com.internal.theming": {
        "modes": {
          "light": { "$value": "{color.white}" },
          "dark": { "$value": "{color.black}" }
        }
      },
      "com.internal.platforms": {
        "ios": { "assetName": "PrimarySurface" },
        "web": { "cssVariable": "--surface-main" }
      }
    }
  }
}
```

Ключевые правила для `$extensions` :

- **Сохранение данных**: Инструменты обязаны сохранять (не удалять) расширения, которые они не понимают, при перезаписи файла.
    
- **Гибкость**: Значения внутри расширений могут быть любыми валидными JSON-данными.
    
- **Семантика**: Расширения не должны изменять базовое значение `$value`, они лишь дополняют его контекстом.
    

## Совместимость с экосистемой инструментов и маппинг данных

Стабильная версия 2025.10 поддерживается широким спектром инструментов, однако уровень реализации спецификации варьируется.

### Style Dictionary v4

Style Dictionary v4 — это основной инструмент трансляции, который теперь имеет первоклассную поддержку DTCG.

- **Уровень поддержки**: Полная поддержка всех 13 типов и 14 цветовых пространств.
    
- **Особенности**: Переход на ES Modules (ESM) и асинхронную модель работы позволяет интегрировать Style Dictionary в современные браузерные среды разработки.
    
- **Ограничения**: Инструмент требует явного переключения на DTCG формат через конфигурацию, так как он сохраняет обратную совместимость с v3 (где использовались ключи `value`, `type` без префикса `$`).
    

### Tokens Studio

Самый мощный плагин для управления токенами в Figma также переходит на стандарт 2025.10.

- **Уровень поддержки**: Высокий, поддержка "Themes" и "Sets" соответствует логике Resolver Module.
    
- **Особенности**: Tokens Studio активно внедряет механизмы автоматической синхронизации с Git-репозиториями, используя DTCG как промежуточный формат.
    

### Figma Variables

Встроенные переменные Figma имеют наиболее ограниченную поддержку спецификации DTCG из-за жесткой внутренней типизации самой Figma.

|**Тип DTCG**|**Тип переменной Figma**|**Ограничения маппинга**|
|---|---|---|
|`color`|**Color**|Figma поддерживает только HEX/RGBA в sRGB. Все расширенные пространства будут аппроксимированы.|
|`number`, `dimension`|**Number**|Единицы измерения (rem, em) теряются, передается только числовое значение.|
|`fontWeight`|**Number** / **String**|Зависит от плагина-конвертера. Прямого маппинга "насыщенность шрифта" в переменные Figma нет.|
|`typography`, `shadow`|**Не поддерживается**|Figma Variables на данный момент не поддерживают композитные типы. Они реализуются через Figma Styles.|

Спецификация DTCG 2025.10 — это не просто документ, а манифест новой эры в разработке интерфейсов. Она позволяет строить по-настоящему адаптивные, технологичные и доступные дизайн-системы, которые живут синхронно в коде и дизайне, минимизируя человеческий фактор и ускоряя вывод продуктов на рынок. Для профессиональных команд внедрение этого стандарта становится необходимым шагом для обеспечения масштабируемости и долговечности их технологического стека.
