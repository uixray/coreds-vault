# Техническое задание на разработку программного комплекса автоматизированной генерации кода Wirecoding AI для среды Figma

## 1. Введение и концептуальные основы проекта

### 1.1. Контекст и обоснование разработки

В современной индустрии разработки цифровых продуктов наблюдается критический разрыв между этапами проектирования интерфейсов (Design) и их технической реализации (Development). Этот процесс, традиционно называемый "Handoff", сопряжен со значительными временными затратами, ошибками интерпретации визуальных макетов и рутинной работой по переносу стилей и структуры в программный код. Существующие решения часто ограничиваются жесткими алгоритмическими трансляторами, которые генерируют семантически некорректный или избыточный код, требующий существенного рефакторинга.

Настоящее техническое задание (ТЗ) инициирует разработку интеллектуального плагина для графического редактора Figma — **Wirecoding AI**. Проект направлен на решение вышеуказанных проблем посредством интеграции передовых генеративных языковых моделей (LLM), способных интерпретировать визуальный контекст и структуру макета подобно опытному frontend-разработчику.

Анализ текущей документации платформы Figma и возможностей современных AI-провайдеров демонстрирует техническую возможность создания инструмента, который не просто транслирует пиксели в код, но и понимает семантику элементов, используя эвристические алгоритмы и возможности компьютерного зрения (Vision API).

### 1.2. Цели и ключевые метрики успеха

Генеральной целью проекта является создание инструментария, сокращающего время верстки типовых интерфейсных компонентов на 60-80% за счет автоматизации написания boilerplate-кода.

**Ключевые задачи:**

1. **Семантическая интерпретация:** Обеспечить преобразование плоской иерархии слоев Figma в семантически верное дерево компонентов (React/Vue/HTML), используя анализ имен слоев и визуальных свойств.
    
2. **Мультимодальность:** Реализовать поддержку ввода как текстового описания (структура JSON), так и растровых изображений макетов для использования моделей с поддержкой Vision (GPT-4o, Claude 3.5 Sonnet, Gemini 1.5 Pro).
    
3. **Агностичность к провайдеру:** Архитектура должна поддерживать горячее переключение между облачными (YandexGPT, OpenAI, Anthropic, Google Gemini) и локальными (Ollama) моделями инференса.
    
4. **Безопасность данных:** Реализация архитектуры BYOK (Bring Your Own Key) с локальным хранением секретов, исключающая компрометацию пользовательских данных.
    

### 1.3. Терминология и сокращения

В документе используются следующие узкоспециализированные термины:

- **Sandbox (Песочница):** Изолированная среда выполнения JavaScript-кода плагина, имеющая доступ к `figma` API, но лишенная доступа к сети и DOM браузера.
    
- **UI Thread (Iframe):** Поток интерфейса плагина, работающий как стандартная веб-страница внутри iframe, имеющий доступ к Browser API (Fetch, WebSocket, LocalStorage), но изолированный от прямой манипуляции документом Figma.
    
- **Origin 'null':** Специфическое состояние источника запроса (Origin) для iframe плагина, которое блокирует стандартные CORS-запросы к большинству публичных API без явного разрешения `Access-Control-Allow-Origin: *`.
    
- **Node (Узел):** Базовый строительный блок документа Figma (Frame, Group, Text, Vector).
    
- **Token Window (Контекстное окно):** Лимит объема информации (в токенах), который модель может обработать за один запрос. Для YandexGPT Pro это 32k токенов , для Gemini 1.5 Pro — до 1-2 млн токенов.

## 2. Архитектура программного комплекса

Разработка плагинов для Figma требует строгого следования архитектурному паттерну, продиктованному соображениями безопасности и производительности платформы. Плагин функционирует как распределенная система из двух изолированных процессов, обменивающихся сообщениями.

### 2.1. Компонентная диаграмма и потоки данных

Архитектура системы разделена на два основных контура: **Sandbox Controller** и **UI Client**.

#### 2.1.1. Sandbox Controller (Backend-like Logic)

Этот компонент выполняется в главном потоке движка Figma (на базе WebAssembly). Он имеет полный доступ к объектной модели документа (DOM Figma), но ограничен в сетевом взаимодействии.

- **Функции:**
    
    - Сканирование выделенных областей (`figma.currentPage.selection`).
        
    - Рекурсивный обход дерева узлов и извлечение свойств (геометрия, стили, эффекты).
        
    - Растрирование узлов в изображения (`exportAsync`) для мультимодальных запросов.
        
    - Управление локальным хранилищем (`figma.clientStorage`) для сохранения настроек и токенов.
        
    - Запись сгенерированных данных обратно в макет (опционально, например, добавление комментариев в код).
        

#### 2.1.2. UI Client (Frontend Logic)

Компонент представляет собой React-приложение, запускаемое в изолированном iframe.

- **Функции:**
    
    - Рендеринг интерфейса управления (выбор модели, ввод промпта, настройки).
        
    - Выполнение сетевых запросов (HTTP/REST) к API AI-провайдеров, так как Sandbox лишен доступа к `fetch` (за исключением ограниченного API).
        
    - Обработка ответов от LLM и отображение сгенерированного кода с подсветкой синтаксиса.
        
    - Управление состоянием приложения (State Management) и валидация форм.
        

#### 2.1.3. Шина данных (Inter-Process Communication)

Взаимодействие между Sandbox и UI осуществляется через асинхронный протокол передачи сообщений.

- **Sandbox -> UI:** `figma.ui.postMessage({ type: 'SELECTION_UPDATED', payload: data })`
    
- **UI -> Sandbox:** `parent.postMessage({ pluginMessage: { type: 'SAVE_SETTINGS', payload: data } }, '*')`.
    

**Важное архитектурное ограничение:** Передаваемые сообщения должны быть сериализуемыми (JSON-safe). Это означает невозможность прямой передачи функций или циклических ссылок.

### 2.2. Сетевая безопасность и обход CORS

Одной из наиболее сложных технических проблем при разработке плагинов Figma является политика **Cross-Origin Resource Sharing (CORS)**. Поскольку плагин работает с `origin: null`, браузер блокирует запросы к любым серверам, которые явно не возвращают заголовок `Access-Control-Allow-Origin: *` или `Access-Control-Allow-Origin: null`.

**Стратегия сетевого взаимодействия:**

1. **Прямое соединение (Direct Connection):**
    
    Используется для API, поддерживающих CORS с wildcard (`*`).
    
    - _Google Gemini API:_ Поддерживает прямые запросы при правильной настройке ключей.
        
    - _Anthropic API:_ Требует использования специального клиентского прокси или разрешения браузера, так как строгие политики безопасности часто блокируют запросы из `null` origin.
        
2. **Локальное проксирование (Local Proxy / Ollama):**
    
    Для работы с локальными моделями через Ollama (`localhost:11434`) требуется специфическая конфигурация среды пользователя, так как стандартный браузерный sandbox блокирует доступ к localhost из публичного ориджина.
    
    - **Решение:** Пользователь должен запустить Ollama с переменной окружения `OLLAMA_ORIGINS="*"`.
        
    - _Mac/Linux:_ `launchctl setenv OLLAMA_ORIGINS "*"`.
        
    - _Windows:_ Установка системной переменной среды.
        
    - Без этой настройки плагин будет получать сетевую ошибку (Network Error) или блокировку CORS.
        
3. **Внешний прокси-сервис (Corsfix / Middleware):** Для API, не поддерживающих настройку CORS (например, некоторые корпоративные инстансы YandexGPT или устаревшие API), в архитектуру может быть включен промежуточный прокси-сервер. Однако, в рамках парадигмы безопасности "Client-Only", использование публичных прокси (типа `corsfix.com` ) не рекомендуется для передачи чувствительных данных (промптов и кода). Рекомендуется предоставить пользователю возможность указать свой URL прокси-сервера.

## 3. Модуль экстракции контекста (Context Engine)

Качество генерации кода напрямую зависит от того, насколько точно и полно плагин сможет описать визуальный макет для языковой модели. Модуль экстракции отвечает за перевод бинарной структуры Figma в текстовое описание (JSON Prompt).

### 3.1. Алгоритм обхода и нормализации данных

Процесс экстракции инициируется при выделении пользователем корневого фрейма.

**Этапы обработки:**

1. **Декомпозиция дерева:** Рекурсивный обход `node.children`. Для каждого узла определяется его роль в верстке.
    
2. **Оптимизация токенов:** Поскольку контекстные окна моделей ограничены (и тарифицируются), JSON-структура должна быть минифицирована.
    
    - Значения по умолчанию (например, `opacity: 1`, `rotation: 0`) исключаются.
        
    - Координаты округляются до целых пикселей (`Math.round`).
        
    - Текстовый контент длиннее 100 символов обрезается (если это "рыбный" текст), но сохраняется маркер длины.
        

**Таблица 3.1. Карта маппинга свойств Figma в CSS/JSON**

|**Свойство Figma**|**JSON Field**|**Интерпретация для LLM**|**Комментарий**|
|---|---|---|---|
|`primaryAxisAlignItems`|`justifyContent`|`flex-start`, `center`, `space-between`|Важно для AutoLayout|
|`counterAxisAlignItems`|`alignItems`|`flex-start`, `center`, `stretch`|Важно для AutoLayout|
|`itemSpacing`|`gap`|`px` value|Отступ между элементами|
|`fills`|`background`|Hex/RGBA|Поддержка градиентов обязательна|
|`strokes`|`border`|`width`, `color`, `style`|-|
|`effects` (DROP_SHADOW)|`boxShadow`|CSS shadow syntax|-|
|`fontName`|`fontFamily`, `weight`|String / Number|Mapping на стандартные веб-шрифты|

### 3.2. Эвристический анализ именования (Naming Conventions)

Согласно исследованиям , именование слоев является критическим фактором для понимания структуры макета. Разработчики и дизайнеры часто используют соглашения, которые плагин должен парсить.

**Правила парсинга имен:**

1. **Семантические теги:** Если слой назван `Button/Primary`, `Input/Search`, плагин должен передать это в промпт как явное указание компонента (`<Button variant="primary">`).
    
2. **Игнорирование мусора:** Имена типа `Frame 134`, `Rectangle 5` должны игнорироваться моделью или заменяться на обобщенные типы (`Container`, `Block`), чтобы не сбивать с толку LLM.
    
3. **Соглашения BEM:** Поддержка распознавания классов, записанных в имени слоя через точку или подчеркивание (например, `card__header`, `btn--large`).
    
4. **Специальные маркеры:** Реализация логики игнорирования слоев, начинающихся с `_` или `.` (например, `_GridOverlay`), которые не должны попадать в финальный код.
    

### 3.3. Визуальный захват (Vision API Support)

Для моделей, поддерживающих мультимодальный ввод (Gemini, Claude 3.5, GPT-4o), плагин должен отправлять не только JSON, но и скриншот компонента.

**Техническая реализация:**

Использование метода `node.exportAsync({ format: 'JPG', constraint: { type: 'SCALE', value: 2 } })`.

- **Сжатие:** Полученный буфер данных (Uint8Array) конвертируется в Base64. Из-за ограничений на размер Payload в некоторых API (например, AWS Bedrock или старые версии Vertex AI), может потребоваться клиентское сжатие изображения (снижение качества JPEG до 0.7) перед отправкой.
    
- **Ограничение размера:** Figma не позволяет экспортировать изображения размером более 4096px по любой стороне. Необходимо реализовать проверку и автоматическое масштабирование вниз (downscaling), если узел превышает лимиты.

## 4. Спецификация интеграции с AI-провайдерами (Gateway Specification)

Система должна реализовывать паттерн "Адаптер" для унификации запросов к различным API. Каждый адаптер отвечает за формирование тела запроса, добавление специфичных заголовков и парсинг ответа.

### 4.1. YandexGPT (Yandex Cloud)

Интеграция с российским облачным провайдером требует особого внимания к механизму аутентификации.

**Аутентификация:** Документация указывает на два метода: IAM-токен и API-ключ. Для плагина (клиентское приложение) использование IAM-токена затруднительно из-за его короткого срока жизни (12 часов) и необходимости сложной логики обновления.

- **Требование:** Использовать API-ключ сервисного аккаунта. Пользователь должен создать сервисный аккаунт, назначить роль `ai.languageModels.user` и выпустить API-ключ.
    

**Параметры модели:**

- **URI:** `gpt://<folder_id>/yandexgpt/latest` (Pro версия для лучшего качества кода) или `gpt://<folder_id>/yandexgpt-lite/latest` (для скорости).
    
- **Контекст:** До 32,768 токенов для Pro-версии 5-го поколения.
    

Формат запроса (JSON Body) :

JSON

```
{
  "modelUri": "gpt://<user_folder_id>/yandexgpt/latest",
  "completionOptions": {
    "stream": true,
    "temperature": 0.2,
    "maxTokens": "8000"
  },
  "messages":
}
```

### 4.2. Google Gemini (Vertex AI / AI Studio)

Google предоставляет мощные модели с огромным контекстным окном, что идеально для генерации целых страниц.

**Аутентификация:** Использование заголовка `x-goog-api-key`. Внимание: использование стандартного `Authorization: Bearer` с API-ключом приведет к ошибке 401.

Safety Settings : Модели Google имеют агрессивные фильтры безопасности. При генерации кода (особенно HTML форм, SVG) возможны ложные срабатывания.

- **Требование:** Явно устанавливать пороги блокировки в `BLOCK_ONLY_HIGH` или отключать их, если это допустимо политиками использования.
    
    - `HARM_CATEGORY_DANGEROUS_CONTENT`: `BLOCK_ONLY_HIGH`
        
    - `HARM_CATEGORY_HATE_SPEECH`: `BLOCK_ONLY_HIGH`
        

JSON Schema Output : Gemini поддерживает режим "JSON Mode" (Structured Outputs). Для плагина это критически важно, чтобы получать не просто текст кода, а структурированный объект:

JSON

```
{
  "componentName": "UserProfile",
  "code": "export const...",
  "dependencies": ["lucide-react", "clsx"]
}
```

Это достигается через параметр `responseMimeType: "application/json"` и передачу схемы в `responseSchema`.

### 4.3. Anthropic (Claude)

Модели Claude 3.5 Sonnet считаются "золотым стандартом" для генерации кода на текущий момент (2025-2026).

Специфика API :

- **Headers:** Обязателен заголовок `anthropic-version: 2023-06-01`.
    
- **Role Alternation:** API требует строгого чередования сообщений `user` и `assistant`. Нельзя отправить два сообщения от `user` подряд.
    
- **System Prompt:** В новых версиях API системный промпт передается как отдельное поле верхнего уровня `system`, а не как сообщение в массиве `messages`.
    

**Обработка ошибок:**

Claude API может возвращать ошибки `overloaded_error` (сервер перегружен). Плагин должен реализовывать стратегию экспоненциальной задержки (Exponential Backoff) для повторных запросов.

### 4.4. Локальные модели (Ollama)

Интеграция с Ollama обеспечивает полную приватность кода.

Техническая реализация :

- **Endpoint:** POST `/api/chat`.
    
- **Image Support:** Передача массива изображений в поле `images` (Base64) внутри объекта сообщения.
    
- **Параметр `format`:** Использование `json` для гарантированного формата ответа.
    
- **Keep-Alive:** Настройка параметра `keep_alive` (например, "5m"), чтобы модель не выгружалась из памяти между запросами, ускоряя итеративную разработку.

## 5. Пользовательский интерфейс и UX

Интерфейс плагина должен быть не только функциональным, но и эстетически интегрированным в среду Figma.

### 5.1. Технологический стек UI

Для разработки интерфейса выбрана библиотека **React**, как стандарт де-факто для плагинов Figma.

- **UI Kit:** Использование **ShadCN/UI** (на базе Radix UI и Tailwind) или **MUI** (Material UI). Анализ трендов 2025 года показывает, что разработчики предпочитают headless-компоненты (Radix, React Aria) с кастомной стилизацией через Tailwind для уменьшения размера бандла и гибкости.
    
- **Code Editor:** Интеграция `react-monaco-editor` или облегченного `react-simple-code-editor` для отображения результата. Важна поддержка подсветки синтаксиса (Syntax Highlighting) и копирования.
    

### 5.2. Сценарии взаимодействия

1. **Настройка ("Setup"):**
    
    - При первом запуске пользователь видит Wizard.
        
    - Шаг 1: Выбор провайдера (Dropdown).
        
    - Шаг 2: Ввод API ключа (Input type="password").
        
    - Шаг 3: Тест соединения (Ping).
        
2. **Генерация ("Workflow"):**
    
    - Пользователь выделяет элементы на холсте.
        
    - Плагин отображает превью выделенных узлов (количество, типы).
        
    - Кнопка "Generate Code" инициирует процесс.
        
    - **Стриминг:** Ответ от модели должен выводиться потоком (char-by-char) в реальном времени, чтобы пользователь видел прогресс и не блокировал интерфейс.
        
3. **Итерация ("Refining"):**
    
    - Чат-интерфейс под сгенерированным кодом.
        
    - Возможность написать: "Сделай кнопку красной" или "Используй Grid вместо Flex".
        
    - Плагин отправляет новый запрос с историей сообщений.

## 6. Управление данными и хранилище

Эффективное управление квотами хранения является критическим аспектом стабильности.

### 6.1. Ограничения ClientStorage

Figma выделяет **5MB** локального хранилища на один плагин (`figma.clientStorage`).

- **Стратегия использования:**
    
    - API-ключи: занимают пренебрежимо мало места.
        
    - История чатов: может быстро заполнить 5MB.
        
    - Кэш настроек: минимальный объем.
        

Алгоритм ротации данных (Eviction Policy) : При достижении лимита (ошибка `quota exceeded`), плагин должен удалять старые записи истории генерации. Рекомендуется хранить не более 10 последних сессий генерации.

### 6.2. Ограничения SharedPluginData

Метод `setSharedPluginData` позволяет сохранять данные внутри самого файла Figma, что делает их доступными для других пользователей файла.

- **Лимит:** 100 кБ на узел (namespace/key).
    
- **Использование:** Хранение метаданных о том, что "этот слой был сгенерирован с помощью Wirecoding AI" и краткой сигнатуры промпта. Не сохранять здесь полный сгенерированный код или историю чата.

## 7. Безопасность и Конфиденциальность

### 7.1. Модель BYOK (Bring Your Own Key)

Плагин не предоставляет собственный бэкенд-прокси для обработки запросов к LLM (кроме опциональных случаев), чтобы избежать передачи ключей через третьи руки.

- **Хранение:** Ключи хранятся в `clientStorage` в исходном виде (так как плагин работает в песочнице на клиенте). Шифрование на клиенте (AES) возможно, но ключ шифрования все равно придется хранить там же или требовать ввод пароля при каждом запуске.
    
- **Рекомендация:** Реализовать опцию "Не запоминать ключ", требующую ввода ключа при каждой сессии для параноидального режима безопасности.
    

### 7.2. Защита от утечек данных

- **Network Access List:** В манифесте плагина (`manifest.json`) должны быть жестко прописаны разрешенные домены (`api.anthropic.com`, `generativelanguage.googleapis.com`, `llm.api.cloud.yandex.net`). Это гарантирует, что даже при компрометации кода (XSS) злоумышленник не сможет отправить данные на произвольный сервер.

## 8. Контроль качества и Тестирование

### 8.1. Стратегия тестирования

1. **Unit Tests (Jest/Vitest):**
    
    - Тестирование функций экстракции: `FigmaNode -> JSON`. Проверка корректности маппинга AutoLayout свойств в Flexbox атрибуты.
        
    - Тестирование утилит именования: Проверка парсинга BEM-классов из названий слоев.
        
2. **Integration Tests:**
    
    - Мокирование ответов API (использование `msw` - Mock Service Worker) для проверки реакции UI на различные коды ответов (200, 401, 429, 500).
        
    - Тестирование сценариев ре-аутентификации при истечении токена.
        
3. **Manual Testing:**
    
    - Проверка работы с большими вложенными фреймами (Deep Nesting).
        
    - Тестирование на различных разрешениях экрана и ОС (Windows/macOS).
        

### 8.2. Обработка ошибок (Error Handling)

Плагин должен предоставлять понятные сообщения об ошибках.

- **401 Unauthorized:** "Ошибка авторизации. Проверьте ваш API ключ в настройках."
    
- **429 Too Many Requests:** "Превышен лимит запросов к AI. Подождите некоторое время."
    
- **Network Error (CORS):** Для Ollama — вывод инструкции по настройке `OLLAMA_ORIGINS`. Для облачных API — проверка интернет-соединения.

### 10.1. Внутренний формат представления узла (Intermediate Node Representation)

Для стандартизации данных перед отправкой в различные LLM используется единый TypeScript интерфейс.

TypeScript

```
type NodeRole = 'container' | 'text' | 'image' | 'button' | 'input' | 'unknown';

interface FigmaContextNode {
  id: string;
  name: string;
  role: NodeRole; // Вычисляется эвристически
  type: SceneNode['type'];
  geometry: {
    x: number;
    y: number;
    w: number;
    h: number;
  };
  layout?: {
    mode: 'NONE' | 'HORIZONTAL' | 'VERTICAL';
    align: string; // "center", "space-between"
    gap: number;
    padding: { t: number; r: number; b: number; l: number };
  };
  styles: {
    fill?: string; // Hex
    stroke?: string;
    radius?: number;
    shadow?: string;
    font?: {
      family: string;
      size: number;
      weight: number;
      align: string;
    };
  };
  children?: FigmaContextNode;
}
```
